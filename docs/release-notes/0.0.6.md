# Release notes for version 0.0.6

## 🎉 New Features

- **Persistent sessions across server restarts**: the session cookie keys are now generated once and
  stored in `cookie.key` inside `data_dir`, so users stay logged in through routine restarts and
  deployments instead of being sent back to the login page
- **Desktop table view for My Bookings**: on large screens, My Bookings now defaults to a compact,
  scannable table (date, item, area, status, on-behalf info, actions) with a tiles/table switch; the
  choice is remembered per browser, and small screens keep the familiar tile view
- **Named on-behalf bookings**: bookings you made for a colleague now show "On behalf of *name*" in
  My Bookings, bookings a colleague made for you show who booked them, and guest bookings show the
  guest's name
- **Application version reporting**: the running version is shown in the user menu, exposed at
  `/api/v1/version`, and printed by the new `sithub version` command
- **Unified colleague booking**: every booking action — item tiles, week mode, the weekly table, and
  the floor plan — opens the same confirmation dialog with a "for me / for a colleague" choice; the
  previously separate inline colleague dropdown on the items page is gone
- **One booking per area and day**: booking a second item in the same area on the same day now asks
  whether to cancel the existing booking and book the new one instead. The same protection applies
  when booking for a colleague: "*name* has already booked *item* on *date*". The swap creates the
  new booking first and cancels the old one only on success, so a failed booking never leaves the
  seat empty

## 🐞 Bug Fixes

- The interactive floor plan now opens for the day or calendar week selected on the page instead of
  always showing today
- Presence avatars in the "today's presence" view are properly centered and use the filled style
- Booker names in the weekly table use the full cell width instead of being truncated by the
  avatar layout
- Dialog buttons with long translated labels (for example the German swap confirmation) wrap onto
  their own line instead of overflowing the dialog

## 🔩 Improvements

- **My Bookings API**: on-behalf bookings now carry `for_user_id` and `for_user_name` attributes and
  guest bookings carry `guest_name`, so clients can attribute seats without guessing from display
  names (additive, JSON:API)
- **Test reliability**: eliminated a race in the live-feed websocket test that intermittently failed
  CI runs, and moved the booking-flow end-to-end tests to the unified confirmation dialog
- **Security scanning**: the unfixable Go advisory GO-2026-5932 (unmaintained `x/crypto/openpgp`,
  never imported by SitHub) is accepted via `.trivyignore` with a documented rationale, keeping the
  Trivy gate green without disabling it
- **Dependency refresh**: vitest toolchain updated to 3.2.7

## ⬆️ Upgrade Notes

- No database migrations required. Rebuild and redeploy as usual.
- On the first start after the upgrade, SitHub creates `cookie.key` (mode `0600`) in `data_dir`.
  Sessions from before the upgrade are invalidated one last time; from then on they survive
  restarts. Back the file up together with `sithub.db` — deleting or replacing it logs every user
  out, and a corrupt file is a hard startup error rather than a silent regeneration (see the
  deployment guide).
- Release binaries report their version automatically. Self-compiled binaries print `dev` unless
  built with `go build -ldflags "-X main.version=<version>" ./cmd/sithub`.
- The My Bookings API gained additive attributes (`for_user_id`, `for_user_name`, `guest_name`);
  no client changes are required.
