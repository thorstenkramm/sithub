# Release notes for version 0.0.5

## 🎉 New Features

- **Consistent item warnings across every booking view**: item warnings now share a single
  presentation and behaviour on item tiles, the weekly table, and the interactive floor plan.
  The weekly table and the floor plan gained a real pre-booking warning confirmation they
  never had before, while the tiles keep their familiar look
- **Uniform, sequential warning confirmation**: booking a warned item shows one consistent
  confirmation dialog everywhere; in week mode, warnings for multiple items are confirmed one
  after another and cancelling any of them aborts the whole booking
- **Floor-plan warning badges**: free items that carry a warning now show an orange warning
  badge centered on the top-right corner of the tile — mirroring the favorite heart — with the
  warning text on hover; booked items continue to show only the booker, never a warning
- **Warning dismissals keyed to the warning text**: "Don't show again" now remembers a
  dismissal per warning text, so if an item's warning is later changed the confirmation
  re-appears instead of staying silently suppressed

## 🔩 Improvements

- **Security hardening**: the server now sends standard HTTP security response headers via
  Echo's Secure middleware (content-type-options, frame-options, XSS-protection, and HSTS
  where TLS is used)
- **Slow-client protection**: HTTP server read/write/idle timeouts and a 2 MB request body
  limit were added to reduce exposure to slow-client and oversized-payload abuse; the avatar
  upload route keeps its larger limit so profile images still work
- **Secure cookies behind a proxy**: a new optional `force_secure_cookies` setting always sets
  the `Secure` flag on session and OAuth state cookies, for deployments that terminate TLS at a
  reverse proxy
- **Expanded security regression tests**: added coverage for the login rate limiter (HTTP 429),
  avatar and floor-plan path-traversal attempts, cookie flags, and the new body limit and
  timeouts
- **Dependency refresh**: updated frontend dependencies including Vite 8, Vue 3.5.39,
  Vuetify 3.12.9, Vue Router 5.1, and vue-i18n 11.4, plus test/lint tooling; npm audit and Trivy
  scans are clean. Major upgrades of Vuetify, vitest, TypeScript, and ESLint are intentionally
  deferred and tracked as backlog candidates

## 🐞 Bug Fixes

- Clicking a cell in the weekly table could open the booking popover and the cancellation
  popover at the same time; only the popover relevant to the clicked cell now appears
- Booking or cancelling from the weekly table no longer flickers the booker avatars or jumps
  the scroll position back to the top — the table now refreshes in place
- Ticking "Don't show again" in the warning dialog no longer aborts an in-progress table
  booking; the dialog stays open until you press Confirm or Cancel
- On the floor plan, the warning badge no longer floated inside the tile; it is now centered on
  the top-right corner, aligned with the favorite heart

## ⬆️ Upgrade Notes

- No database migrations required. Rebuild and redeploy as usual.
- The server now enforces HTTP timeouts and a 2 MB request body limit. Normal API requests are
  unaffected and the avatar-upload route keeps its larger limit; make sure any reverse-proxy
  body-size limit still accommodates avatar uploads.
- New optional `force_secure_cookies` (default `false`; override with the `--force-secure-cookies`
  flag or the `SITHUB_MAIN_FORCE_SECURE_COOKIES` environment variable): enable it when TLS is
  terminated by a reverse proxy so session and OAuth cookies always get the `Secure` flag.
  Existing setups are unchanged with it left off.
- Standard security response headers are now sent by default; if your reverse proxy already sets
  similar headers, review for duplication.
- Frontend and Go dependencies were updated; no configuration changes are required.
